    OWASP SQL Injection Security Patterns

    SlowQL provides comprehensive SQL injection detection based on OWASP Top 10 vulnerabilities and industry best practices. This guide covers all 35 SQL injection patterns implemented in Sprint 1.

    üîí Overview

    SQL injection remains one of the most critical web application security risks. SlowQL's OWASP Security Detector Suite provides:

    • 35 comprehensive detection patterns covering all major SQL injection techniques
    • Multi-vector coverage including boolean-based, union-based, time-based, and error-based attacks
    • NoSQL injection support for MongoDB and other document databases
    • Advanced evasion detection including comment-based and encoding obfuscation
    • Real-world attack patterns based on current threat intelligence

    üéØ Attack Vector Categories

    1. Boolean-based Blind SQL Injection (11 patterns)

    Detects inference-based attacks where attackers extract information through boolean logic.

    AND 1=1 Patterns

    -- Detected patterns:
SELECT * FROM users WHERE id = 1 AND 1=1
SELECT * FROM products WHERE name LIKE '%test%' AND 1=1

    Rule: SEC-INJ-002 - Boolean-based Blind SQLi (AND)

    OR 1=1 Patterns

    -- Detected patterns:
SELECT * FROM users WHERE id = 1 OR 1=1
SELECT * FROM products WHERE active = 1 OR 1=1

    Rule: SEC-INJ-003 - Boolean-based Blind SQLi (OR)

    Substring Extraction

    -- Detected patterns:
SELECT * FROM users WHERE SUBSTRING(password,1,1)='a'
SELECT * FROM accounts WHERE MID(username,1,1)='admin'

    Rule: SEC-INJ-004 - Boolean-based Blind SQLi (Substring)

    ASCII/CHAR Comparison

    -- Detected patterns:
SELECT * FROM users WHERE ASCII(SUBSTRING(password,1,1))>64
SELECT * FROM data WHERE CHAR(65)=SUBSTRING(name,1,1)

    Rule: SEC-INJ-005 - Boolean-based Blind SQLi (ASCII)

    Length-based Enumeration

    -- Detected patterns:
SELECT * FROM users WHERE LENGTH(password)>5
SELECT * FROM data WHERE LENGTH(username)=8

    Rule: SEC-INJ-006 - Boolean-based Blind SQLi (LENGTH)

    EXISTS Function Exploitation

    -- Detected patterns:
SELECT * FROM users WHERE EXISTS(SELECT 1 FROM admin WHERE id=1)
SELECT * FROM products WHERE EXISTS(SELECT * FROM hidden WHERE secret=1)

    Rule: SEC-INJ-007 - Boolean-based Blind SQLi (EXISTS)

    CASE Statement Logic

    -- Detected patterns:
SELECT * FROM users WHERE CASE WHEN 1=1 THEN 1 ELSE 0 END=1
SELECT * FROM data WHERE CASE WHEN SUBSTRING(pwd,1,1)='a' THEN 1 ELSE 0 END=1

    Rule: SEC-INJ-008 - Boolean-based Blind SQLi (CASE)

    IN Function Enumeration

    -- Detected patterns:
SELECT * FROM users WHERE id IN ('1','2','admin')
SELECT * FROM data WHERE name IN ('test','admin','user')

    Rule: SEC-INJ-009 - Boolean-based Blind SQLi (IN)

    BETWEEN Range Testing

    -- Detected patterns:
SELECT * FROM users WHERE id BETWEEN 1 AND 100
SELECT * FROM data WHERE LENGTH(name) BETWEEN 1 AND 20

    Rule: SEC-INJ-010 - Boolean-based Blind SQLi (BETWEEN)

    Time-based Boolean Attacks

    -- Detected patterns:
SELECT * FROM users WHERE id=1 AND SLEEP(5)
SELECT * FROM products WHERE active=1 AND (SELECT COUNT(*) FROM users)>0

    Rule: SEC-INJ-011 - Boolean-based Blind SQLi (Time-based)

    2. Union-based SQL Injection (5 patterns)

    Detects data extraction attacks using UNION statements.

    Basic UNION SELECT

    -- Detected patterns:
SELECT username FROM users UNION SELECT password FROM admins
SELECT id FROM products UNION SELECT id FROM orders

    Rule: SEC-INJ-012 - Union-based SQLi (SELECT)

    UNION ALL Complete Extraction

    -- Detected patterns:
SELECT * FROM users UNION ALL SELECT * FROM admins
SELECT name FROM products UNION ALL SELECT name FROM services

    Rule: SEC-INJ-013 - Union-based SQLi (UNION ALL)

    System Table Targeting

    -- Detected patterns:
SELECT * FROM users UNION SELECT * FROM sys.tables
SELECT * FROM products UNION SELECT * FROM information_schema.tables
SELECT name FROM users UNION SELECT name FROM pg_tables

    Rule: SEC-INJ-014 - Union-based SQLi (System Tables)

    NULL-based Column Enumeration

    -- Detected patterns:
SELECT * FROM users UNION SELECT NULL,NULL,NULL
SELECT id FROM products UNION SELECT NULL,NULL,NULL,NULL

    Rule: SEC-INJ-015 - Union-based SQLi (NULL)

    Multiple UNION Operations

    -- Detected patterns:
SELECT * FROM users UNION SELECT * FROM admins UNION SELECT * FROM logs
SELECT name FROM products UNION SELECT name FROM services UNION SELECT name FROM inventory

    Rule: SEC-INJ-016 - Union-based SQLi (Multiple)

    3. Time-based Blind SQL Injection (5 patterns)

    Detects timing-based information extraction attacks.

    SLEEP Function Attacks

    -- Detected patterns:
SELECT * FROM users WHERE id=1 AND SLEEP(5)
SELECT * FROM products WHERE active=1 AND SLEEP(3)

    Rule: SEC-INJ-017 - Time-based Blind SQLi (SLEEP)

    BENCHMARK CPU-intensive Attacks

    -- Detected patterns:
SELECT * FROM users WHERE id=1 AND BENCHMARK(1000000,MD5(1))
SELECT * FROM products WHERE active=1 AND BENCHMARK(500000,SHA1(1))

    Rule: SEC-INJ-018 - Time-based Blind SQLi (BENCHMARK)

    PostgreSQL pg_sleep

    -- Detected patterns:
SELECT * FROM users WHERE id=1 AND pg_sleep(5)
SELECT * FROM data WHERE active=1 AND pg_sleep(3)

    Rule: SEC-INJ-019 - Time-based Blind SQLi (pg_sleep)

    SQL Server WAITFOR DELAY

    -- Detected patterns:
SELECT * FROM users WHERE id=1 AND WAITFOR DELAY '00:00:05'
SELECT * FROM products WHERE active=1 AND WAITFOR DELAY '00:00:03'

    Rule: SEC-INJ-020 - Time-based Blind SQLi (WAITFOR)

    Heavy Query Timing Attacks

    -- Detected patterns:
SELECT * FROM users WHERE id>0 AND (SELECT COUNT(*) FROM products WHERE id>0)>10
SELECT * FROM data WHERE active=1 AND (SELECT COUNT(*) FROM logs WHERE id>0)>100

    Rule: SEC-INJ-021 - Time-based Blind SQLi (Heavy Query)

    4. Error-based SQL Injection (5 patterns)

    Detects information disclosure through error message manipulation.

    CONVERT Function Exploitation

    -- Detected patterns:
SELECT * FROM users WHERE id=CONVERT(varchar,'test')
SELECT * FROM products WHERE name=CONVERT(int,'invalid')

    Rule: SEC-INJ-022 - Error-based SQLi (CONVERT)

    CAST Function Attacks

    -- Detected patterns:
SELECT * FROM users WHERE name=CAST('invalid' AS int)
SELECT * FROM data WHERE id=CAST('text' AS datetime)

    Rule: SEC-INJ-023 - Error-based SQLi (CAST)

    MySQL UPDATEXML Exploitation

    -- Detected patterns:
SELECT * FROM users WHERE id=UPDATEXML(1,'/test','invalid')
SELECT * FROM products WHERE name=UPDATEXML(1,'//test','payload')

    Rule: SEC-INJ-024 - Error-based SQLi (UPDATEXML)

    MySQL EXTRACTVALUE Exploitation

    -- Detected patterns:
SELECT * FROM users WHERE id=EXTRACTVALUE(1,'//test')
SELECT * FROM data WHERE name=EXTRACTVALUE(1,'/invalid/path')

    Rule: SEC-INJ-025 - Error-based SQLi (EXTRACTVALUE)

    Division by Zero Errors

    -- Detected patterns:
SELECT 1/0 FROM users
SELECT * FROM products WHERE id=1/0
SELECT COUNT(*)/0 FROM data

    Rule: SEC-INJ-026 - Error-based SQLi (Division by Zero)

    5. Second-order SQL Injection (3 patterns)

    Detects stored malicious code that executes later.

    Stored Procedure Execution

    -- Detected patterns:
INSERT INTO logs (message) VALUES ('test') AND EXEC('sp_configure')
UPDATE users SET password='new' WHERE id=1 AND EXECUTE('xp_cmdshell')

    Rule: SEC-INJ-027 - Second-order SQLi (Stored)

    Trigger and Procedure Creation

    -- Detected patterns:
CREATE TRIGGER test ON users AFTER INSERT AS EXEC('malicious_code')
ALTER PROCEDURE sp_test AS EXEC('harmful_command')

    Rule: SEC-INJ-028 - Second-order SQLi (Trigger)

    Malicious View Creation

    -- Detected patterns:
CREATE VIEW admin_view AS SELECT * FROM users UNION SELECT admin,pwd FROM admins
ALTER VIEW user_data AS SELECT * FROM secret_table

    Rule: SEC-INJ-029 - Second-order SQLi (View)

    6. NoSQL Injection (2 patterns)

    Detects injection attacks in NoSQL databases.

    MongoDB JavaScript Injection

    -- Detected patterns (MongoDB-style):
db.users.find({$where: function() { return this.password == "' + userInput + '" }})
db.products.find({name: {$ne: null}}) // Operator injection

    Rule: SEC-INJ-030 - NoSQL Injection (MongoDB)

    NoSQL Operator Manipulation

    -- Detected patterns:
db.data.find({$or: [{admin: true}, {$where: "this.password=='"+input+"'"}]})
db.users.find({$and: [{role: "user"}, {name: {$ne: "admin"}}]})

    Rule: SEC-INJ-031 - NoSQL Injection (Operator)

    7. Advanced Evasion Techniques (4 patterns)

    Detects sophisticated attack methods and bypass techniques.

    Combined Attack Vectors

    -- Detected patterns:
SELECT * FROM users WHERE id=1 AND 1=1 UNION SELECT admin FROM users
SELECT * FROM products WHERE name='test' OR 1=1 AND SLEEP(5)

    Rule: SEC-INJ-032 - Advanced SQLi (Combined)

    Comment-based Evasion

    -- Detected patterns:
SELECT * FROM users WHERE id=1 /* AND 1=1 */ OR 1=1 --
SELECT name FROM products /* comment */ UNION SELECT name FROM admins

    Rule: SEC-INJ-033 - SQLi (Comment Evasion)

    Encoding and Obfuscation

    -- Detected patterns:
SELECT * FROM users WHERE name=0x61646d696e UNION SELECT 0x70617373
SELECT * FROM products WHERE id=CHAR(49) UNION SELECT CHAR(97,100,109,105,110)

    Rule: SEC-INJ-034 - SQLi (Encoding Evasion)

    Database-specific Function Exploitation

    -- Detected patterns:
SELECT * FROM users WHERE DBMS_LOCK.SLEEP(5)
SELECT name FROM products WHERE DBMS_OUTPUT.PUT_LINE('exfiltrated')
SELECT * FROM data WHERE SP_EXECUTESQL('SELECT * FROM admins')

    Rule: SEC-INJ-035 - SQLi (Advanced DB Functions)

    üõ°Ô∏è Protection Recommendations

    1. Parameterized Queries (Primary Defense)

    # ‚ùå Vulnerable
query = "SELECT * FROM users WHERE id = " + user_input

# ‚úÖ Secure
query = "SELECT * FROM users WHERE id = %s"
cursor.execute(query, (user_input,))

    2. Input Validation

    • Whitelist acceptable input patterns
    • Validate data types and lengths
    • Sanitize special characters

    3. Least Privilege

    • Use database accounts with minimal permissions
    • Separate read and write operations
    • Implement role-based access controls

    4. Error Handling

    • Disable detailed error messages in production
    • Implement generic error responses
    • Log security events for monitoring
